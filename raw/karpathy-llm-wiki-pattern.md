## 核心理念：Wiki 而非 RAG

大多数人用 LLM 处理文档的方式是 RAG：上传文件，LLM 在查询时检索相关块，生成答案。这没有积累——每次问问题，LLM 都在从头重新发现知识。

这里提出的不同：不是在查询时从原始文档检索，而是让 LLM 增量构建并维护一个持久化 wiki——一个结构化、相互链接的 markdown 文件集合，位于你和原始来源之间。

**关键差异**：wiki 是一个持久的、复利增长的产物。交叉引用已经在那里。矛盾已经被标记。综合已经反映了你读过的一切。每增加一个来源，每问一个问题，wiki 都变得更丰富。

"I rarely touch the wiki directly. It's the domain of the LLM." — Andrej Karpathy

## 三层架构

### 第一层：原始来源（Raw sources）
你精心挑选的源文档集合。文章、论文、图片、数据文件。**不可变**——LLM 从中读取但从不修改。这是你的真相来源。

### 第二层：Wiki
LLM 生成的 markdown 文件目录。摘要、实体页面、概念页面、比较、概述、综合。LLM 完全拥有这一层——创建页面、在新来源到来时更新、维护交叉引用、保持一切一致。你读它，LLM 写它。

### 第三层：Schema（配置文件）
告诉 LLM wiki 如何结构化、约定是什么、摄入来源/回答问题/维护 wiki 时遵循什么工作流的文档（如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md）。这是关键配置文件——让 LLM 成为有纪律的 wiki 维护者，而不是通用聊天机器人。你和 LLM 随时间共同演化这个文件。

## 三个核心工作流

### Ingest（摄入）
将新来源放入原始集合并告诉 LLM 处理。示例流程：LLM 阅读来源，与你讨论关键要点，在 wiki 中写摘要页面，更新索引，更新整个 wiki 中相关的实体和概念页面，并在日志中追加条目。**单个来源可能触及 10-15 个 wiki 页面。**

### Query（查询）
针对 wiki 提问。LLM 搜索相关页面、阅读它们，综合带引用的答案。重要洞见：好的答案可以作为新页面归档回 wiki。你问的比较、分析、发现的关联——这些有价值的内容不应该消失在聊天历史里。

### Lint（健康检查）
定期请 LLM 对 wiki 进行健康检查：页面间的矛盾、新来源已取代的陈旧声明、没有入链的孤儿页面、被提及但缺乏自己页面的重要概念、缺少的交叉引用、可以通过网络搜索填补的数据空白。

## 两个特殊文件

### index.md（内容导向）
wiki 中所有内容的目录——每个页面都有链接、一行摘要和可选元数据。LLM 在每次摄入时更新它。回答查询时，LLM 先读索引找相关页面，再深入。这在中等规模（~100 个来源，~数百个页面）效果出奇的好，避免了基于嵌入的 RAG 基础设施需求。

### log.md（时间导向）
只追加的发生记录——摄入、查询、lint 过程。每个条目以一致前缀开始（如 `## [2026-04-02] ingest | 文章标题`），日志可以用简单的 unix 工具解析。

## 推荐工具
- **Obsidian Web Clipper**：将网页文章转换为 markdown，快速获取来源
- **qmd**：本地 markdown 搜索引擎，具有混合 BM25/向量搜索和 LLM 重排序，有 CLI 和 MCP Server
- **Obsidian 图视图**：查看 wiki 形状——什么连接什么，哪些页面是枢纽，哪些是孤儿
- **Marp**：基于 markdown 的幻灯片格式
- **Dataview**：运行对页面 frontmatter 的查询

## 本质洞见

**维护知识库的繁琐部分不是阅读或思考，而是账务工作。** 更新交叉引用、保持摘要最新、注意新数据何时与旧声明矛盾、在数十个页面间保持一致性。人类放弃 wiki 是因为维护负担增长比价值更快。LLM 不会感到无聊，不会忘记更新交叉引用，并且可以在一次操作中触及 15 个文件。

人类的工作是策划来源、指导分析、提出好问题、思考这一切意味着什么。LLM 做其他一切。

## 历史渊源
这个想法与 Vannevar Bush 的 Memex（1945）在精神上相关——一个具有文档间关联路径的个人、精心策划的知识存储。Bush 无法解决的部分是谁来做维护。LLM 处理了这个问题。
