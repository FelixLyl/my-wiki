# Agent 系统的三类熵增防线

自主 Agent 系统的核心工程问题不是"如何让它做得更多"，而是"如何防止它的失控积累"。三个看似无关的设计模式——Agent 安全体系、GStack 流程结构、幂等防重——在这个维度上高度同构：它们各自在不同层面对抗同一件事：**熵增**。

## 三个层面的熵

**权限熵**：Agent 持续运行，权限边界自然蔓延。每多接入一个工具、每多赋予一个 Skill，潜在攻击面就扩大一格。OpenClaw 安全体系的应对是零信任 + 操作可逆：不信任任何输入、最小权限、高危操作强制人工介入。这是**对权限熵的遏制**。

**流程熵**：多 Agent 并行运行，任务之间的上下文隔离，信息孤岛和遗漏环节自然积累。GStack 的应对是流程结构化：`Think → Plan → Build → Review → Test → Ship → Reflect`，每个 Skill 输出喂给下一个，强制信息流通。这是**对流程熵的遏制**。

**内容熵**：知识库被反复读写，同一素材多次处理、LLM 每次理解略有不同，内容逐渐失真积累。幂等防重模式的应对是 LEDGER + 内容哈希：同一文件只处理一次，修改才触发重处理。这是**对内容熵的遏制**。

## 统一框架

三类防熵机制的共同结构：

| 维度 | 熵的来源 | 防熵机制 | 核心原语 |
|------|---------|---------|---------|
| 安全 | 权限蔓延 / 外部注入 | 零信任 + 操作可逆 | 人在回路 |
| 流程 | 上下文孤岛 / 遗漏环节 | 流程结构化 + 信息传导 | 强制接力 |
| 内容 | 重复处理 / 理解漂移 | 幂等账本 + 哈希检测 | 增量优先 |

每种机制都依赖一个**外部记录系统**来对抗 Agent 的短期记忆局限：安全依赖审计日志，流程依赖 design doc 传递上下文，知识库依赖 LEDGER 账本。

## 推论

一个没有防熵机制的 Agent 系统，会在运行时间足够长后自然退化为混乱状态：权限越界、流程断链、内容重复失真。三类防熵不是独立的工程优化，而是同一个系统属性的三个投影——**可信 Agent 系统的本质是熵的管理**。

"增量永远优于全量"（幂等模式）在权限和流程层面同样成立：每次只做最小必要的变更，比全面重跑更安全、更可追溯、更不容易出错。
