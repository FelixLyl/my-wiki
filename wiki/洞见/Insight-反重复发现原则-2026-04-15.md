# 反重复发现原则

三个看似无关的机制——CLAUDE.md 配置方法论、Wiki vs RAG 的选择，以及 Prompt Caching——在底层共享同一个设计意图：**让已知的东西不必再被重新发现一遍**。

## 三篇文章的碰撞

[[CLAUDE-md配置方法论]] 的核心价值是声明式配置：将项目规范"内化"为 AI 行为。4 条黄金准则写入 CLAUDE.md 后，每次 Agent 启动都无需重新教会它这些规则。这是规范知识的"固化"。

[[Wiki-vs-RAG]] 指出了两种知识存储模式的本质差异：RAG 没有积累性，每次查询都在重新发现同样的知识；Wiki 是复利增长的，交叉引用和矛盾标记已经在库中等着。Karpathy 的表达更直接——"好的分析消失在聊天历史里"是一种知识浪费。

[[Prompt-Caching降本]] 从经济学角度定量化了同一原则：静态上下文重复计算是对 Token 的浪费。缓存机制让"不变的部分"只付一次钱，差量部分按需计费。

## 统一的底层逻辑：认知经济学视角

这三个机制都在解决"重复支付认知成本"的问题，只是发生在不同层次：

| 层次 | 机制 | 重复成本的形式 |
|------|------|--------------|
| 行为规范层 | CLAUDE.md 声明式配置 | 每次任务重新解释规则 |
| 知识积累层 | Wiki（而非 RAG） | 每次查询重新发现关联 |
| 计算资源层 | Prompt Caching | 每次请求重复计算静态前缀 |

三层堆叠构成一个完整的"反重复发现"系统：规范写入 CLAUDE.md → 知识沉淀到 Wiki → 上下文缓存到 Prompt Cache。从人类时间，到 AI 推理时间，到 API 费用，每一层都在执行同一个优化动作。

## 推论：一切"内化"都是缓存

这个视角有一个更激进的推论：所谓"学习"，无论是人的学习、AI 的配置，还是知识库的维护，本质上都是把"需要每次重新推导的东西"变成"可以直接调用的结果"。

内化的经济学逻辑是：**重复发现的成本大于存储的成本时，存储就有意义。**

这个阈值因载体不同而不同：人的记忆有容量限制（选择性记忆），Wiki 有维护成本（需要持续整理），Prompt Cache 有时效限制（5 分钟有效期）。系统设计者的任务是在这些约束下，找到值得"固化"的知识边界。

## 对个人知识库设计的启示

1. **索引是知识的缓存前缀**：[[Wiki-vs-RAG]] 中提到的 `_index.md`（每个页面一行摘要）正是把"哪些知识在哪里"这件事缓存成一个可快速读取的结构，而非每次全量扫描。

2. **CLAUDE.md 写的越好，Wiki 就越轻**：如果 AI 行为规范已经内化到配置层，知识库就不需要记录"应该怎么做"，只需要记录"世界是什么样的"。两者各司其职，避免冗余。

3. **Prompt Caching 的边界即系统知识的边界**：什么内容值得放进静态前缀（缓存对象），什么内容是动态输入（每次不同），这个分割线本质上是在回答"哪些知识是稳定的、可共享的"。这与 Wiki 中 maturity 字段的设计逻辑（authoritative → 稳定可引用）同构。

## 尚待验证的问题

- 当知识更新频率超过某个阈值时，固化的成本可能高于重复发现的成本。这个阈值如何量化？
- CLAUDE.md、Wiki 和 Prompt Cache 三层之间的一致性如何维护？若 Wiki 更新了，Cache 失效是自动的，但 CLAUDE.md 的行为规范是否需要对应调整？
- "反重复发现"原则在多 Agent 场景中的复杂度：多个 Agent 各自维护知识缓存，如何避免同一知识在不同 Agent 记忆中产生分歧？（参见 [[分布式Agent与知识孤岛]]）
