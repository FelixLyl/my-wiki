# 安全体系的自进化闭环

静态安全规则（OWASP Top 10、零信任清单、供应链扫描）在 AI Agent 系统中面临一个根本缺陷：攻击面以 Agent 能力的扩张速度演化，而防护规则以人工审查的速度更新。两者之间的速度差，正是安全漏洞滋生的土壤。

## 三篇文章的碰撞点

[[OpenClaw-Agent安全体系]] 构建了一套分层防护：供应链安全（skill-vetter/guard）+ 运行时拦截（ops-guardrails）+ 零信任原则。但这套体系本质上是静态的——规则由人写入，不会从 Agent 自身的运行经验中更新。

[[GStack-虚拟工程团队]] 的 `/cso` 角色执行 OWASP Top 10 + STRIDE 威胁建模，设置 8/10+ 置信度门槛。这是流程内嵌安全（security-in-sprint），而非事后审查。gstack 的核心设计思想是**流程化结构消除信息孤岛**——每个环节的输出喂给下一个。

[[自学习复盘模式]] 的 self-evolving-skill 机制给出了关键补丁：Agent 可以读取报错日志、自动修改 Skill 代码以适配新环境。这是自动化维护逻辑，但原文没有将其扩展到安全规则本身。

## 隐藏的组合：安全即进化循环

将三者组合，可以描绘一个尚未存在、但逻辑上自洽的架构：

```
Sprint 执行（gstack流程）
    → 触发 /cso 安全评审 → 记录安全决策（通过/拒绝/修改）
    → 安全失败模式结构化为知识卡片（自学习复盘）
    → 卡片写入 Agent 记忆库
    → 下次 /cso 执行时，主动检索历史失败模式
    → 防护规则随运行经验动态更新
```

这个循环的关键是**安全决策的记忆化**：不只是"这次通过了 OWASP 检查"，而是"这种 Prompt Injection 模式在上次被拦截，这次遇到变体时仍应触发警报"。

## 实践意义

1. **安全知识卡片的可信度分级**：人工审查确认的安全案例高可信，Agent 自推断的低可信——这与自学习复盘模式的约束一致，防止错误学习固化成错误规则。

2. **gstack 的 `/investigate` 铁律**（无调查不修复，3 次失败后停止）可以扩展为安全版本：无法解释的安全告警，不得静默通过，必须记录原因供后续学习。

3. **零信任原则遇到进化机制的张力**：零信任要求"不信任任何输入"，而自学习机制本质上是"从历史经验中建立信任"。两者并不矛盾，但边界需要明确：对**已知模式的信任度可以提升**，对**未知输入的信任度永远从零开始**。

## 尚未解决的问题

- 安全知识卡片的"遗忘机制"：旧的安全模式过期后如何清除，避免对已修复漏洞保持过度警觉？
- 自进化安全规则本身是否成为攻击面？对手可以通过构造历史案例来污染安全知识库（对抗性记忆注入）。
- 在 gstack 的 sprint 流程中，哪个环节最适合触发安全学习？`/retro`（复盘）是候选，但需要显式设计。

安全体系从静态规则集进化为动态学习系统，是 Agent 系统走向生产成熟度的必经路径。目前这个闭环在开源工具中尚无完整实现，是一个值得深入设计的方向。
