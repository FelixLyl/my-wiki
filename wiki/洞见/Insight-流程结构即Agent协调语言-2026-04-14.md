# 流程结构即 Agent 协调语言

三个表面无关的话题——GStack 的 sprint 流程、Agent 安全体系的人在回路、wiki 做梦机制的三阶段设计——指向同一个结论：**多 Agent 系统的真正协调语言不是消息传递，而是流程结构本身。**

## 碰撞轨迹

[[GStack-虚拟工程团队]] 的 23 个专家角色为何能并行不混乱？核心答案不是"角色划分清晰"，而是 `think → plan → build → review → test → ship → reflect` 的线性 sprint 流程。每个 Agent 知道：自己的输入从哪个阶段来，输出去哪个阶段。没有这条流水线，23 个 Agent 同时开工只会产生 23 个相互矛盾的方案。

"流程化结构（think→plan→build→review→test→ship）是并行不混乱的关键——每个 Agent 知道自己在哪个阶段，该停在哪里等待人工决策。"

[[OpenClaw-Agent安全体系]] 的人在回路机制做的是同一件事：在 Agent 决策链中，用结构（高危操作触发审批）而非规则（不要做坏事）来协调 Agent 与人之间的边界。规则可以被绕过或遗忘，但流程节点是强制的。

[[做梦机制]] 的三阶段设计（夜间巡逻→灵感碰撞→健康检查）同样如此：单个 Agent 执行一组看似不相关的任务，不乱的原因是任务之间有明确的阶段边界和前置条件。做梦机制不允许健康检查修改文章主体——这个约束不是靠 Prompt 叮嘱，而是写在工作流阶段定义里。

## 模式抽象

**流程结构即协调语言（Workflow as Coordination Protocol）**

> 在多 Agent 或单 Agent 多任务场景中，用明确的阶段顺序和边界约束取代模糊的角色规则。
> 每个阶段定义：输入来源、可执行操作、输出目标、停止条件。
> 这使 Agent 的局部自主性得以安全扩展，而不依赖 Agent 的全局判断能力。

这与编译器理论中的"文法约束语义"类似：语法的合法性不靠编写者的自觉，而靠结构本身排除非法路径。

## 三角对比

| 系统 | 协调问题 | 流程解法 | 边界约束 |
|------|---------|---------|---------|
| GStack | 23 个 Agent 并行方案冲突 | think→plan→build→review 线性流水线 | 人工决策节点强制暂停 |
| Agent 安全体系 | Agent 越权或误操作 | 高危操作触发审批阶段 | 运维护栏拦截，非规则约束 |
| 做梦机制 | 单 Agent 多任务相互干扰 | 巡逻→碰撞→健康检查三段式 | 禁止健康检查修改文章主体 |

## 实践含义

- **设计多 Agent 系统时**：先定义 sprint 阶段，再分配角色，而非反过来。角色是流程的执行者，不是流程的定义者。
- **写 Agent 约束时**：尽量写成"阶段 X 不能执行 Y 操作"，而非"你应该避免 Y"。前者是结构约束，后者是道德劝诫。
- **知识库设计中**：做梦机制的"绝不修改已有内容主体"是阶段约束，让"创造性阶段"和"维护阶段"不互相污染。

多 Agent 系统的复杂性管理，本质上是流程工程学，而非 Prompt 工程学。
