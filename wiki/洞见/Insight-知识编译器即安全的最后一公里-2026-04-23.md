# 知识编译器即安全的最后一公里

GStack 的 `/cso` 是代码层面的安全守门人，OpenClaw 安全体系是运行时的护栏，而 Karpathy 式的个人知识库编译器——看似与安全无关——实际上承担了第三层：**认知层面的安全积累**。

## 三者的隐性连接

GStack 将安全内嵌为流程节点（OWASP Top 10 + STRIDE），强制每个 sprint 必须过 `/cso` 关卡，而不是靠工程师的记忆。这是一种**程序性防御**：安全检查不依赖人的状态，依赖流程位置。

OpenClaw 安全体系的零信任原则把信任问题外化为制度：不信任任何输入，不信任 Agent 自身判断，所有高危操作插入人工审批节点。这是**结构性防御**：把安全边界刻进系统拓扑。

Karpathy 的 wiki 模式解决的问题不同：他把零碎的教训、决策、模式沉淀为持久化的结构化知识，让 LLM 成为维护者。这是**记忆性防御**：安全知识不随人员流动或会话终止而消失。

"I rarely touch the wiki directly. It's the domain of the LLM." — Andrej Karpathy

## 最后一公里问题

程序性防御和结构性防御都有一个共同的弱点：它们保护的是已知的威胁面。未知的、模糊的、经验性的威胁——比如"这个 Skill 的写法看起来没问题但有历史先例表明类似模式会出问题"——需要的不是规则，而是积累的认知。

这正是知识编译器的补位点。每次 skill-vetter 扫描出一个新模式，每次 ops-guardrails 拦截一次误操作，这些事件如果只停留在日志里，下一个 Agent 实例仍然是零经验。但如果编译进 wiki——作为 `pattern` 或 `decision` 类型的条目——下一个实例从一开始就携带这份经验。

## 闭环：流程 × 结构 × 记忆

三层防御合并后构成一个闭环：

```
流程防御（GStack /cso）
  → 发现新威胁模式
  → 结构防御（OpenClaw guardrails）更新规则
  → 记忆防御（wiki compiler）编译为经验条目
  → 下一个 Agent 的 prompt context 携带此经验
  → 流程防御更加精准
```

缺少任何一层，闭环就断：纯流程防御不能自学习；纯结构防御不能处理边缘情况；纯记忆积累没有强制执行点。

## 实践推论

在 OpenClaw 部署中，这意味着：不仅要安装 skill-vetter 和 ops-guardrails，还需要一个主动的"安全经验入库"机制——把每次安全事件、每次 vetter 的发现、每次 guardrails 的触发，通过 wiki-compiler 的 `decision` 工作流编译成持久知识。

这不是文档工作，是安全体系的认知层基础设施。
