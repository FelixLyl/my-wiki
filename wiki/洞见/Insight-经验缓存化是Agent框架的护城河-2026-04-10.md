# 经验缓存化是 Agent 框架的护城河

Hermes Agent 在两个月内积累接近三万 Star，证明开源社区能够快速复刻一个"功能完整"的 Agent 框架。功能层面的竞争壁垒正在消失。但有一样东西无法被 fork：**积累的经验**。

## 碰撞的三篇文章

- [[自学习复盘模式]]：Agent 通过记录错误知识卡片，实现行为层面的自我进化
- [[Prompt-Caching降本]]：静态前缀缓存命中率最高，动态内容后置；"前缀不变"是降本关键
- [[Hermes-Agent]]：开源社区两个月复刻主流能力，功能壁垒快速消解

## 核心洞见

Prompt Caching 的技术原理揭示了一个更深层的架构哲学：**越老的知识越应该在前面**。系统提示中的行为规范、工具定义这些"已知经验"被前置为不可变前缀，才能稳定命中缓存。

这与自学习复盘模式的逻辑天然契合：被纠正的错误知识卡片，一旦写入，就不再修改（仅追加新卡片）。这类"仅追加、永不修改"的经验层，恰好满足 Prompt Cache 的前缀不变性要求。

由此推论：**一个持续运行的 Agent 框架，其经验知识库本身可以被结构化为缓存友好的多层前缀**——

| 层次 | 内容 | 更新频率 | 缓存适宜度 |
|------|------|---------|----------|
| L0 基础规范 | 安全红线、工具定义、格式规范 | 几乎不变 | ⭐⭐⭐⭐⭐ |
| L1 历史经验 | 错误知识卡片、已解决的 Bug 模式 | 仅追加 | ⭐⭐⭐⭐ |
| L2 项目上下文 | 当前项目架构、近期决策 | 低频更新 | ⭐⭐⭐ |
| L3 当前任务 | 本次对话、实时工具输出 | 每轮变化 | ❌ 不缓存 |

Hermes Agent 可以复刻 L0，但复刻不了 L1 和 L2——那是某个具体部署实例在真实使用中沉淀的经验，不在代码仓库里。

## 护城河的含义

对于 Agent 框架的竞争而言，这意味着：

1. **每一次用户纠正都是资产**：纠正写入 L1 经验层，下次同类错误的 Token 成本更低，准确率更高
2. **时间创造差异化**：运行越久的实例，经验层越厚，总拥有成本（Token 成本 + 错误成本）越低
3. **迁移成本是经验层的厚度**：用户从一个 Agent 框架迁移到另一个，本质上是丢弃了积累的 L1/L2 层经验

换言之，开源竞争者对抗的是 L0 的功能壁垒；而真正的护城河在 L1 和 L2——它们无法被 fork，只能被时间积累。

## 尚待验证

- 多层缓存前缀在实际 API 中的可行性（当前 Prompt Cache 主要缓存单一前缀，分层缓存是否有技术限制？）
- 经验层过厚时的负面效应：过时知识卡片的干扰、缓存失效后的成本骤升
- 不同 Agent 框架在经验可移植性上的设计差异（是否有标准化格式？）
