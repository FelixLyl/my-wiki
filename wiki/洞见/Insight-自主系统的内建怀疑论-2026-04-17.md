# 自主系统的内建怀疑论

三个看似无关的设计——Agent 安全体系的零信任原则、GStack 的 `/investigate` 铁律、以及做梦机制的健康检查——在底层逻辑上指向同一个模式：**把不信任自身的视角内建到系统运行流程中**。

## 发现的隐藏连接

[[OpenClaw-Agent安全体系]] 的零信任原则核心是"不信任任何输入"和"人在回路"：高权限 Agent 对外部世界保持警觉，高危操作前强制插入人工确认节点。

[[GStack-虚拟工程团队]] 的 `/investigate` 命令有一条铁律："无调查不修复，3 次失败后停止"。这是对 Agent 自身输出的不信任——即使修复方案看起来合理，也必须先彻底调查根因，而不是让系统自信地反复尝试同一个错误思路。`/cso` 安全审计以 8/10+ 置信度门槛过滤，也体现了对自身生成内容的二次质疑。

[[做梦机制]] 的健康检查阶段则是对知识库自身的不信任：主动扫描死链、孤儿页面、超长文章，而不是假设知识库结构始终健康。zhangpelf V2 进一步引入"零幻觉"约束——遇到知识盲区时不强行填充，而是生成警告追踪卡，把不确定性显性化。

## 核心模式：内建怀疑论（Built-in Skepticism）

这三个系统都在做同一件事：**将外部审查员的视角制度化，嵌入到系统自身的执行流程中**，而不是依赖外部的事后检查。

| 系统 | 内建怀疑论的具体形式 | 怀疑的对象 |
|------|------|------|
| Agent 安全体系 | 高危操作前强制人工审批 | 外部输入 + 自身行为 |
| GStack `/investigate` | 3 次失败后停止 + 无调查不修复 | 自身的修复方案 |
| 做梦健康检查 | 定期扫描死链孤儿 + 不确定性显性化 | 知识库的自身状态 |

## 反面案例：什么系统缺少这一模式

缺乏内建怀疑论的自主系统往往表现为：
- Agent 重复尝试同一错误方案，无终止条件
- 知识库积累沉默的错误，直到问出错误答案时才被发现
- 代码通过 CI 但在生产崩溃（gstack 的 `/review` 的存在动机）

## 推论：怀疑论需要独立视角

内建怀疑论要起效，怀疑者必须与被怀疑者有某种隔离。OpenClaw 安全体系建议"独立审计员实例"，GStack 用 `/codex` 引入第二个模型（Claude + OpenAI 双重评审），做梦机制在空闲时以"非写作模式"扫描知识库。共同点是：**同一个系统在不同的上下文/模式/实例下审视自己**，而不是用生产思路审计生产状态。

这与人类认知中"快思考/慢思考"的切换有结构上的相似：系统 1（执行）和系统 2（审查）需要刻意分离，才能互相制衡。
