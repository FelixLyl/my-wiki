# 错误记忆的缓存友好编码

[[自学习复盘模式]]、[[Prompt-Caching降本]]、[[Andrej-Karpathy]] 三者碰撞产生一个未被明确提出的工程洞见：**Agent 的错误知识卡片，本身就应该被设计成缓存友好的格式**。

## 碰撞点

[[自学习复盘模式]] 描述了 Agent 如何将被纠正的错误行为结构化为知识卡片，写入记忆库供后续任务调用。这些卡片通常以系统提示的形式注入 Agent 的上下文。

[[Prompt-Caching降本]] 揭示了一个关键约束：缓存只对稳定的前缀有效。若系统提示频繁变动（新增卡片、修改措辞），缓存失效，成本上升。

Karpathy 的 wiki 理念（[[Andrej-Karpathy]]）补充了第三视角：知识应该"积累"而非"替换"。wiki 的核心价值是复利增长——新内容追加，旧内容保留，整体结构收敛趋于稳定。

## 洞见

将三者联系起来，可以推出一个未被显式设计的工程规范：

**错误知识卡片应采用"仅追加、永不修改"的编码策略**，以保持系统提示前缀的最大稳定性。

具体而言：
- 已有卡片内容冻结，只追加新卡片
- 卡片排序固定（按时间或按主题哈希），不因新增重排
- 每张卡片格式一致（字段顺序、措辞模板固定），减少 Token 级别的 diff

这样，系统提示中的"错误记忆区"能持续命中 Prompt Cache，同时知识本体按 Karpathy 式复利积累。

## 引申：记忆即代码

这一视角将 Agent 的记忆工程与软件工程类比：

| 软件工程 | Agent 记忆工程 |
|---------|-------------|
| 追加写日志（Append-only log） | 仅追加错误卡片 |
| 不可变构建产物（Immutable artifact） | 冻结已有卡片内容 |
| 语义版本控制 | 知识卡片 maturity 级别（stub → authoritative） |
| 缓存友好的静态资源 | 前缀稳定的系统提示 |

当 Agent 的记忆被视为"可缓存的不可变数据结构"而非"可随时修改的便签"，成本和知识质量可以同时优化。

## 待验证

- 实际测量：同等卡片数量下，"仅追加冻结"策略 vs "动态修改"策略的缓存命中率差异
- 边界条件：卡片数量增长到多少时，单次系统提示过长，反而需要拆分压缩？
- 与 [[Agent记忆持久化]] 的树状摘要方案如何协作：低层是 append-only 卡片，高层是定期蒸馏的摘要树？
